# Requirements: <功能名>

把功能的边界和关键选择说清楚。格式自由——用段落或列表都行，但这几件事要覆盖：

- **范围**：这个 feature 做什么、包含什么。尽量小，和 roadmap 里这个 phase 的目标一一对应
- **范围外**：明确不做什么。被砍掉的功能如果写在 roadmap 的后续 phase 里，在这里用 `（见 Phase N）` 标注——证明它是「以后再干」而不是「忘了」
- **行为**：从使用者视角描述正常流程、空状态和失败路径。不写代码，只说「会看到什么」「会发生什么」
- **决策**：记录实现上的关键选择以及原因——为什么 id 用整数而不是 UUID，为什么文件放这个路径，等等。如果涉及数据结构，用表格或文字描述均可
- **背景**：语气要求、已有的模式和约定、来自 `mission.md` 和 `tech-stack.md` 的约束、本 feature 对利益相关方诉求的回应
- **未决问题**：还没想清楚但会影响实现的问题，带上暂定的处理方式
